# 金丝雀系统架构现状与开发上下文报告

**范围：** `src/modules/canary/`（`contracts.py`、`strategy.py`、`probes_sannysoft.py`、`dashboard.py`）及其与 `MasterDispatcher`、`PrismAPI`（`bridge.py`）的集成。  
**用途：** 为后续「分步开发指令」提供结构上下文与断层清单。

---

## 1. 系统核心骨架与工作流

金丝雀被实现为 **与主站爬取共用调度器与 Crawlee/反爬栈** 的「合成站点任务」，而非独立进程。

1. **入口（UI → 桥接）**  
   `PrismAPI.run_canary_checkup`（`src/app/bridge.py`）在调度器空闲时：调用 `dashboard.reset_for_new_run()`、写入金丝雀独立日志缓冲区、组装 `task_info.is_canary=True` 与 `strategy_settings.target_urls=DEFAULT_CANARY_SEED_URLS`，再通过 `asyncio.run_coroutine_threadsafe` 投递 `dispatcher.run_with_config(...)`。

2. **策略路由**  
   `MasterDispatcher._select_strategy`（`src/app/dispatcher.py`）在 `mode=="site"` 且 `is_canary` 为真时实例化 `CanaryMockStrategy`，而非 `SiteCrawlStrategy`。

3. **运行时行为（`CanaryMockStrategy`）**  
   - 继承 `SiteCrawlStrategy`，复用父类的 Crawlee 启动、代理/挑战器等基础设施。  
   - **显式跳过生产账本：** 覆盖 `_create_task_record` / `_update_task_status`，避免 `tasks`、审计中心等 DB 被探测流量污染。  
   - **收窄爬虫回调：** `_register_crawlee_handlers` 仅挂 `default_handler` 与可选的 `failed_request_handler`，不注册 NEED_CLICK 等会触发真实下载/交互的路径。  
   - **每页处理：** `default_handler` → `_canary_page_handler_core`：先尝试与挑战页检测协同；再按 URL 分流 **Sannysoft 真实探针** 或 **其它种子的占位断言**；最后 `_bump_progress()` 更新 `dashboard` 进度百分比。

4. **看板读出**  
   前端 `UI/canary.js` 轮询 `fetch_canary_dashboard` → `_build_canary_dashboard_payload` → `dashboard.build_payload`，合并 **内存象限状态** 与 **调度器占用语义**（`idle` / `running` / `locked`）。

## 2. 模块协同与设计模式

| 模块 | 职责 | 模式/约定 |
|------|------|-----------|
| `contracts.py` | 与前端轮询字段对齐的 **JSON 形状类型**（TypedDict） | 契约层，无运行时逻辑 |
| `dashboard.py` | **进程内、线程安全** 的象限行 + 进度条 | 单例式全局状态 + `threading.Lock`；`build_payload` 为只读组装 |
| `probes_sannysoft.py` | 在 **Playwright Page** 上 `evaluate` 抽取 bundle，再 **纯函数** 合成 `(item_id, state, desc)` 列表 | 采集（副作用）与判定（纯函数）分离；`apply_*` 委托 `dashboard.set_quadrant_group` |
| `strategy.py` | **编排：** 何时跑探针、何时写占位、如何映射 URL→象限 | 模板方法变体：继承站点策略骨架，替换页级语义；URL 分流 |

**数据契约流转（Sannysoft 路径）：**

```
Crawlee context.page
  → run_sannysoft_identity_probe(page, settings)
       → _extract_page_bundle (evaluate + 超时)
       → build_sannysoft_probe_updates(bundle, expected_profile)
  → apply_sannysoft_probe_updates(updates)
       → set_quadrant_group("identity", …) / set_quadrant_group("hardware", …)
  → snapshot 由 fetch_canary_dashboard 读出
```

**非 Sannysoft 路径：** 同一 handler 内根据 `_quadrant_for_url` 将种子 URL 与 `_CANARY_QUADRANT_GROUPS` 对齐，用 `page.title()` + HTTP 状态等 **占位断言** 批量 `set_quadrant_group`。

## 3. 核心逻辑完成度

### 3.1 已完成且已贯通 UI

- **内存看板：** 默认四象限骨架、`reset` / `set_progress` / `set_quadrant_group` / `mark_all_failed` / `build_payload` 完整，且与 `contracts.CanaryDashboardDict` 一致。  
- **调度语义：** `dispatcher.is_canary_run_active()` 与 `build_payload` 中的 `system_state` 已打通。  
- **Sannysoft 探针（身份 + 硬件四行）：**  
  - 采集：`page.evaluate` 注入大块 `_SANNYSOFT_EXTRACT_JS`，覆盖 navigator/screen/viewport、表格行 WebDriver、字体数量与 Hook 痕迹、Canvas/WebGL/WebAudio 等。  
  - 判定：`build_sannysoft_probe_updates` 将结果映射到 `identity_locale`、`viewport_fit`、`webgl_vendor`、`canvas_audio`，状态含 `pass` / `warn` / `fail`。  
  - 写入：`strategy` 在识别 `bot.sannysoft.com` / `sannysoft.com` 主机名时调用探针并 `apply_sannysoft_probe_updates`，**已对接 dashboard**，前端轮询即可看到更新。  
- **指纹期望（可选）：** 当 `settings.stealth.use_fingerprint` 为真时，用 `FingerprintGenerator` 生成期望，与页内 UA/分辨率等做严格或宽松比对（见 `_ua_strict_match`、`_screen_compatible`）。

### 3.2 部分完成 / 占位

- **network 象限**（`tls_ja3`、`http_headers`、`webrtc_leak`）与 **combat 象限**（`cf_shield`、`cdp_automation`、`behavior_score`）：当前仅为 **「有标题且 HTTP&lt;400 → pass」类占位**，无 TLS/JA3、报文、WebRTC、CF 盾、CDP、行为评分等 **真实探针或外部数据源**。  
- **Sannysoft 页内数据：** JS 抽取了 `userAgentOld` 表格行，但 Python 侧 **未参与** `build_sannysoft_probe_updates` 的合成逻辑（抽取冗余或预留未用）。

### 3.3 文档与实现轻微漂移

- `PrismAPI.fetch_canary_dashboard` 的 docstring 仍写「占位数据；探针层接入后替换」——与当前 **已接入内存探针态** 不符，属于注释滞后，易误导后续读者。

## 4. 关键数据结构定义片段（`contracts.py`）

以下为金丝雀看板 API 与 UI 对齐的核心 TypedDict（完整定义见源文件）：

```10:31:src/modules/canary/contracts.py
class QuadrantItemDict(TypedDict):
    id: str
    label: str
    state: str
    desc: str


class QuadrantsDict(TypedDict):
    network: List[QuadrantItemDict]
    identity: List[QuadrantItemDict]
    hardware: List[QuadrantItemDict]
    combat: List[QuadrantItemDict]


class CanaryDashboardDict(TypedDict):
    """fetch_canary_dashboard 返回体。"""

    system_state: str
    current_engine: str
    progress_percent: int
    quadrants: QuadrantsDict
```

`dashboard._default_quadrants()` 中各 `item_id` 与 `strategy._CANARY_QUADRANT_GROUPS`、`probes_sannysoft` 中 `_IDENTITY_ITEM_IDS` / `_HARDWARE_ITEM_IDS` **必须保持同名对齐**（当前一致）。

## 5. 未完成 / 断层与代码位置

### 5.1 功能断层（产品语义）

| 区域 | 现状 | 说明 |
|------|------|------|
| Network 三行 | 占位 | 无真实 TLS/JA3、HTTP/IP、WebRTC 检测实现 |
| Combat 三行 | 占位 | 无 CF 挑战、CDP、行为评分等实现 |
| 与「核心引擎」的闭环 | 弱 | 结果仅进内存看板 + 日志；未见写入审计中心、策略回写、持久化报告等（金丝雀刻意零 DB，若产品要历史对比需另设计出口） |

### 5.2 代码级标记

- **`strategy.py` 中 `pass`：** 出现在 `except` 块内（如 `_bump_progress`、挑战失败分支），属 **有意吞异常**，不是未实现占位函数。  
- **`probes_sannysoft.py`：** 无 `TODO`；`userAgentOld` 仅在前端抽取对象中出现，Python 未消费（逻辑断点：数据已采未用）。  
- **`__init__.py`：** 仅说明从 `strategy` 导入以避免循环依赖，无额外 API。

### 5.3 依赖与风险点（非 TODO，但影响「完成度」）

- **Sannysoft DOM：** 表格 `findRow("WebDriver (New)")` 等依赖页面结构；改版会导致 `_webdriver_row_passes` 解析失败并按失败处理（设计已考虑，但运维上需回归）。  
- **`apply_sannysoft_probe_updates`：** 内部 `from src.modules.canary.dashboard import set_quadrant_group` 为 **延迟导入**，用于打破 `dashboard ↔ probes` 潜在环，后续新增模块时勿改为顶层互引。  
- **进度条语义：** `_canary_total = min(len(seeds), len(_CANARY_QUADRANT_GROUPS))`，与「每处理一个种子 URL 一次 `_bump_progress`」一致；Sannysoft 单页更新多行仍只 bump 一次（符合「按 URL 计数」而非「按检测项计数」）。

## 7. 小结（给路线图制定者）

- **骨架已成：** 调度注入 → 专用策略 → Crawlee 页回调 → 内存看板 → pywebview 轮询 的 **端到端通路已通**。  
- **深度探针仅覆盖 Sannysoft 相关的 identity + hardware 四行；network + combat 仍为占位。**  
- **下一步自然切块：** 为 network/combat 增加独立 `probes_*.py`（或统一注册表），在 `strategy._canary_page_handler_core` 中按 URL/主机名分流调用，并继续通过 `set_quadrant_group` 写入同一 `dashboard`；若需持久化或引擎反馈，再在 bridge 层增加显式出口，避免破坏「零 DB 账本」约定。
