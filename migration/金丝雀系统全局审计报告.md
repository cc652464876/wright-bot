## 1. 执行摘要

金丝雀子系统已形成可运行的闭环：**Bridge 投递合成任务 → Dispatcher 选型 `CanaryMockStrategy` → `SiteRunner` 模板保证 `teardown/cleanup` → 内存看板 + `fetch_canary_dashboard` 供前端轮询**。探针层对常见异常普遍做了降级（`fail`/`warn` 或三行兜底），主进程因单页探针异常而崩溃的风险较低。

主要问题集中在三类：（1）**父类 `SiteCrawlStrategy.run()` 吞掉 `crawler.run()` 异常**与金丝雀子类 **`set_progress(100)` 的语义组合**，会导致「进度 100% + 部分象限仍为 idle / 未反映失败」的**严重语义背离**；（2）**默认 Crawlee 并发为 16**，与「每种子 URL 对应单一象限、顺序推进」的隐含假设叠加重试/多回调时，可能出现**进度提前封顶与描述非确定性**；（3）**重定向后 `request.url`/当前页 URL 与种子主机不一致**时的象限归因偏差。

以下按要求的 🟢 / 🟡 / 🔴 分级列出；每条均给出**文件路径、行号与推导逻辑**。

## 3. 🟡 潜在隐患（不立刻崩溃，但需警惕）

### 3.1 默认高并发与「象限单行」模型的张力

- **位置：** Crawlee 工厂使用 `settings.get_effective_max_concurrency()`（`src/modules/site/strategy.py` 约 410–415 行）；`get_effective_max_concurrency` 在 `max_concurrency=="auto"` 时默认为 **16**（`src/config/settings.py` 557–568 行）。
- **推导：** 金丝雀仅 4 个种子 URL，但在高并发与 **请求重试/重复回调** 下，同一象限的 `default_handler`/`failed_handler` 可能被触发多次；`_bump_progress` 仅单调增加计数（`strategy.py` 248–250 行），`set_progress` 使用 `min(100, …)`（`dashboard.py` 62–65 行），可能出现 **进度提前到 100% 而 Crawlee 仍在运行**、或 **同一象限多轮写入以最后一次为准** 的非确定性（非崩溃，**结果可重复性差**）。

### 3.2 重定向与当前 URL 对象限归因

- **位置：** `_canary_page_handler_core` 使用 `current_url = request.url`（`strategy.py` 271–273 行）；`_quadrant_for_url` / `_is_sannysoft_probe_url` 均基于该 URL（253–258、285–288、307 行）。
- **推导：** 若导航终点与种子主机不一致（跳转 CDN、地区跳转等），可能 **未走 Sannysoft 专用分支** 或 **象限与业务预期不符**；属于**环境依赖型误判**，而非纯代码逻辑笔误。

### 3.3 `failed_request_handler` 少数路径未推进进度

- **位置：** `_canary_failed_request_core` 在 `request is None` 时直接 `return`（`strategy.py` 374–376 行），不调用 `_bump_progress`。
- **推导：** 依赖 `CanaryMockStrategy.run` 末尾的 `set_progress(100)`（136–138 行）仍可把进度拉满；若同时存在其它未覆盖路径与父类异常语义问题（见 🔴），**会加剧「进度与分项状态不一致」的观感**。

### 3.4 `bridge.run_canary_checkup` 与策略内双重 `reset_for_new_run`

- **位置：** `bridge.run_canary_checkup` 在投递任务前调用 `reset_for_new_run()`（`src/app/bridge.py` 638–641 行）；`CanaryMockStrategy.run` 再次 `reset_for_new_run()`（`strategy.py` 134–135 行）。
- **推导：** 若在两次重置之间前端轮询，可能看到**短暂全 idle**；功能上可接受，但属于**多余 I/O 与语义噪音**（技术债，见第 6 节）。

### 3.5 宽泛 `except Exception` 与调试可见性

- **位置：** `strategy.py`、`probes_network.py`、`probes_combat.py`、`probes_sannysoft.py` 多处 `except Exception`（示例：`strategy.py` 139–142、211、256、263、280、293、313、342、351、359、368、386 行）。
- **推导：** 多数路径会写日志或映射到 `fail`，**并非静默 `pass`**；但宽泛捕获可能掩盖 **KeyboardInterrupt / SystemExit** 以外的编程错误，长期维护时建议逐步收窄类型（属规范与可维护性，见第 6 节）。

### 3.6 Bridge / Dispatcher 中与金丝雀相邻的「吞异常」

- **位置：** `bridge.set_stealth_engine` 中 `evaluate_js` 失败时 `except Exception: pass`（`src/app/bridge.py` 614–624 行）；`MasterDispatcher._log` 对 UI 回调 `except Exception: pass`（`src/app/dispatcher.py` 427–431 行）。
- **推导：** 与金丝雀看板数据无直接契约破坏，但会导致**前端静默失败**；若未来金丝雀窗口依赖同一模式扩展，需复用同一风险意识。

## 4. 🔴 严重缺陷（逻辑上与「体检完成」语义严重背离 / 高优先级修复）

### 4.1 父类吞掉 `crawler.run()` 异常，子类仍执行 `set_progress(100)`

| 项 | 说明 |
|----|------|
| **位置 A** | `src/modules/site/strategy.py` **256–264** 行：`try: await crawler.run(seeds)` 的 `except Exception` 记录日志并 `await self._fsm_transition(CrawlerState.ERROR, …, swallow=True)`，**不向外重新抛出**。随后方法继续执行排空、`STOPPED` 等（269–293 行），**正常返回**。 |
| **位置 B** | `src/modules/canary/strategy.py` **136–138** 行：`CanaryMockStrategy.run` 在 `await super().run()` 成功后**无条件** `set_progress(100)`。 |
| **推导** | 当 Crawlee 引擎在 `crawler.run` 层抛出异常时，**FSM 已进入 ERROR，但 `super().run()` 对调用方仍视为「正常返回」**，金丝雀侧仍会 **把进度条置为 100%**。同时，`mark_all_failed` 仅在 **136–142 行** `CanaryMockStrategy.run` 自身捕获到异常时调用；该路径**不会因父类已吞掉的 crawler 异常而触发**。若失败未映射到 `failed_request_handler` / `default_handler` 的看板写入，可能出现 **progress_percent=100 而多象限仍为 idle 或旧状态** —— 对「体检是否完成、是否成功」构成 **阻断级别的产品语义错误**（不一定是进程崩溃）。 |

### 4.2 「体检完成」与分项状态可能长期不一致（与 4.1 叠加）

| 项 | 说明 |
|----|------|
| **位置** | `src/modules/canary/strategy.py` **132–138** 行（`_canary_total` 与 `set_progress(100)`）；**248–250** 行（`_bump_progress` 仅按 handler 触发次数递增）。 |
| **推导** | 若部分 URL **从未进入** `default_handler`（引擎早退、队列未消费等），`_bump_progress` 可能不足以反映「四象限均已执行」；在 4.1 场景下仍可能被 **136–138 行** 统一打上 **100%**，加剧 **「满进度 + 未检项」** 的误导性。 |

## 6. 技术债（可维护性 / Pythonic）

1. **`CanaryDashboardDict` 运行时校验：** `build_payload` 使用 `cast(QuadrantsDict, quads)`（`dashboard.py` 129 行），不在运行时校验每项是否含 `id/label/state/desc`；当前默认骨架保证键存在，若未来动态增删行易与 `contracts.py` 漂移。
2. **`set_progress` 内 `global` 声明：** `set_progress` 在 `with _lock` 内声明 `global _progress_percent`（`dashboard.py` 62–65 行），功能正确但可读性略差；可改为在锁外单点更新或封装小函数以减少心智负担。
3. **金丝雀专用并发策略未显式化：** 未在 `CanaryMockStrategy` 或 bridge 组装配置中**显式**将 `max_concurrency` 固定为 1 或文档化「依赖串行语义」，与引擎默认 16 并发存在**隐式假设缺口**（见 🟡 3.1）。

## 附录：架构复核与修正决策（ADR）

本附录记录对前文 🔴 4.1 与看板并发模型的**复核结论**与**后续实现约束**，作为架构决策留痕（Architecture Decision Record，轻量形式）。**本节不替代具体代码实现**，实现时须结合单测与联调验证。

### A. 关于 🔴 4.1 的修复决策（成功判定与 OCP）

**背景：** `SiteCrawlStrategy.run()` 在 `crawler.run()` 抛出异常时会经 FSM 转入 `CrawlerState.ERROR`，但随后仍会经 `STOPPING` 落到 **`CrawlerState.STOPPED`** 并正常返回（见正文 §4.1 引用）。因此**仅凭任务结束时的当前状态是否为 `ERROR` 无法区分**「引擎曾失败但已被收尾」与「全程成功」。

**决策：**

1. **不修改**父类 `SiteCrawlStrategy` 中针对 `crawler.run()` 的异常捕获与吞没行为，以避免改变主站抓取线路的既有语义与回归面，符合**开闭原则（OCP）**与风险隔离。
2. **金丝雀侧**在 `await super().run()` **返回之后**，通过 `CrawlerStateManager.get_history()` 回溯**本次任务生命周期内**是否出现过 **`new_state == CrawlerState.ERROR`** 的转移记录（历史项形如 `(timestamp, old_state, new_state, reason)`，见 `src/engine/state_manager.py` 中 `transition_to` / `get_history` / `reset` 的契约）。
3. **判定语义（实现约束）：** 若历史中出现过 `ERROR`，则视为**本次合成任务在状态机意义上未按成功路径完成**（与正文 🔴 4.1 对齐）：应禁止无条件 `set_progress(100)` 冒充完成，并应执行看板级收敛（例如 `mark_all_failed` 或等价策略，具体文案与是否区分「引擎失败 / 种子为空」等产品规则在编码阶段敲定）。若历史中**未**出现 `ERROR`，则允许在其它条件满足时将进度收敛为 100%。

**依据与边界：**

- `reset()` 会在新任务进入 `_execute_task` 时清空 `_history`（`state_manager.py`），故**单次 `get_history()` 快照对应当前调度任务**，一般不会混入上一次任务的转移（前提是任务入口始终先 `reset`，与现有 `MasterDispatcher` 流程一致）。
- 同一任务内若因其它原因（例如种子 URL 为空）也曾进入 `ERROR`，历史中同样会出现 `ERROR`；是否一律视为「体检失败」须与产品语义一致，必要时可在实现层结合 `reason` 或转移序列做细化，**本 ADR 只锁定「禁止仅看终态、应查历史」这一架构方向**。

### B. 关于 `dashboard.py` 跨线程锁的保留决策

**背景：** 金丝雀看板由 **Crawlee / Playwright 异步回调**（运行于 **asyncio 事件循环线程**）写入，由 **`fetch_canary_dashboard` → `build_payload` → `snapshot_quadrants_progress`** 等路径读出；前端经 **pywebview** 轮询时，Bridge 方法往往在 **GUI 线程**执行（见 `bridge.py` 中对跨线程调度的说明）。

**决策：**

1. **`src/modules/canary/dashboard.py` 中继续保留 `threading.Lock`**，用于保护 `_quadrants`、`_progress_percent` 的读改写与 `copy.deepcopy` 快照，保证多线程环境下的**原子性与可见性**。
2. **禁止**在未建立完整「GUI 线程 ↔ 事件循环」桥接的前提下，将上述锁**盲目替换为 `asyncio.Lock`**：GUI 线程上的同步轮询无法安全、`await` 异步锁，易引入**错误假设、卡死或绕过锁的并发访问**；若未来统一改为「仅在事件循环内读写看板」，再评估迁移方案并单独立项。
3. **性能原则：** 保持当前「短临界区 + 深拷贝快照」模式；避免在持锁期间执行可能阻塞的 I/O 或跨线程等待。
